# Phase 1: Investigation（調査・問題理解）

## 目的

OSSのissueを多角的に調査し、問題の本質と影響範囲を構造化された形で理解する。

## サブエージェント戦略

### 1.1 GitHub Context 取得（URL入力時）

`Task` ツールで `subagent_type: "ctx:github"` エージェントを呼び出す:

```
プロンプト: "{owner}/{repo}#{number} のIssue情報を取得して。
本文、コメント、ラベル、関連PR、Discussionを含めて構造化して返して"
```

返却される構造化コンテキスト（`=== GITHUB_CONTEXT ===` ブロック）から以下を抽出:
- Issue 本文と再現手順
- コメントでの追加情報やメンテナーの反応
- ラベル（bug, enhancement, help wanted 等）
- 関連 PR（過去の修正試行）
- 参照されているファイルパスや関数名

### 1.2 独立並列調査（視点ベース、最大4エージェント）

`Task` ツールで `subagent_type: "Explore"` を最大4並列で起動する。

**核心ルール**: 各agentは他のagentの結果を知らない状態で独立に調査する。これにより確証バイアスを防止する。

**Agent A: symptom-trace（症状からの逆追跡）**
- Issue で報告された症状（エラーメッセージ、異常な挙動）をキーワードに Grep で検索
- コードパスを症状から逆方向に辿り、最初に異常が発生する箇所を特定
- コールチェーンの追跡と問題箇所の周辺コードを Read で確認
- テスト状況の把握（テストディレクトリ構造、テストフレームワーク、既存カバレッジ）

**Agent B: structural-analysis（構造的弱点分析）**
- Issue の報告内容に依存せず、問題箇所周辺の構造的な脆弱点を分析
- 型の整合性、状態管理パターン、エラーハンドリングの欠落を検出
- 依存関係（import/require 先）と影響範囲（下流コード）を Grep で特定
- コーディングパターンの不整合や設計上の問題を検出

**Agent C: historical-context（変更履歴からの推定）**
- `git log` / `git blame` で問題箇所の変更履歴を調査
- 最近の変更で壊れた可能性、リグレッションの検出
- 過去の関連 PR やコミットメッセージから意図を読み取る
- CI 設定（`.github/workflows/`, `Makefile` 等）からテスト実行コマンドを特定

**Agent D: environmental-factors（環境要因）— 条件付き起動**
- 外部依存（ライブラリ、API、ランタイム）が関与する場合のみ起動
- 依存ライブラリのバージョン差異、設定ファイルの問題、ランタイム環境固有の挙動を調査
- package.json / go.mod / Cargo.toml 等からバージョン情報を確認

注意: Issue の内容から問題が明確に局所的な場合は、Agent A + B の2並列で十分。4並列は複雑な問題や原因が不明確な場合に使用する。

**各agentの出力スキーマ**:

各agentは以下の形式で仮説を返す:

```yaml
hypothesis:
  claim: "XXXが原因である"
  evidence:
    - file: "src/parser.ts:45"
      description: "エスケープ処理が欠落している"
      strength: direct_code  # direct_code(3) / test(3) / git_history(2) / doc(1) / inference(0)
  confidence: high / medium / low
  assumptions:
    - "入力にマルチバイト文字が含まれる場合にのみ発生"
```

### 1.3 N-of-1ルール（仮説数の保証）

Phase 1.2 の結果を集約後、仮説が1つしかない場合は追加調査を強制する:

```
IF count(hypotheses) < 2:
  追加agent起動: Task(Explore) with prompt:
    "既に特定された仮説「{claim}」以外の原因を意図的に探索せよ。
     この仮説が正しい場合でも、副次的な要因や隠れた前提条件を探せ。
     異なる視点（例: 仮説がロジックエラーなら型・設定・環境の視点）から調査せよ"
```

これにより、アンカリングバイアス（最初の仮説に引きずられる）を防止する。

### 1.4 仮説-反証サイクル

Phase 1.2〜1.3 で出揃った仮説群に対し、反証調査を実行する。

```
FOR EACH hypothesis IN hypotheses:
  反証agent = Task(Explore) with prompt:
    "仮説「{hypothesis.claim}」が間違いだと仮定する。
     以下の証拠をコードベース内から探して報告せよ:
     1. この仮説の前提条件が成立しないケース
     2. この仮説では説明できない症状や挙動
     3. この仮説と矛盾する既存テストの存在や実行結果
     見つからなかった場合は、探索した範囲と方法を報告せよ"
```

反証結果の評価:
- **反証成功**: 仮説を棄却し、棄却理由を記録
- **反証失敗**: 仮説を強化（ただし「探索範囲が不十分」の可能性も注記）
- **部分反証**: 仮説を修正（条件付きで成立する形に再定義）

### 1.5 証拠強度スコアリングと矛盾検出

#### 証拠強度スコア

```yaml
scoring:
  direct_code_evidence: 3   # コードから直接読み取れる（行番号引用あり）
  test_evidence: 3           # テスト結果が裏付ける
  git_history_evidence: 2    # 変更履歴が裏付ける
  documentation_evidence: 1  # ドキュメントに記載あり
  inference_only: 0          # 推論のみ（コード証拠なし）
  contradicted: -3           # 反証により否定された
```

各仮説の証拠スコアを合計し、ランキングを作成する。

#### 矛盾検出マトリクス

仮説間の関係を分類する:
- **排他的**: 仮説Aが正しければBは成立しない
- **両立可能**: AとBが同時に成立しうる（複合原因）
- **因果関係**: Aが根本原因でBは症状
- **無関係**: 独立した問題

### 1.6 ハルシネーション防止チェック

出力を確定する前に以下を検証する:

1. **ソース実在確認**: 出力に含まれるファイルパス・関数名を `Read` / `Grep` で再確認
2. **コード内容照合**: 引用したコードの内容が実際と一致するか `Read` で検証
3. **API仕様の裏取り**: ライブラリの挙動に言及する場合、`WebSearch` で公式ドキュメントを取得し照合
4. **バージョン整合性**: 参照したドキュメントのバージョンがプロジェクトの依存バージョンと一致するか確認
5. **推論と事実の区別**: 各主張に証拠タグ（direct_code / inference 等）を付与。inference のみの主張は「推論」と明記

詳細なチェック項目: [bias-checklist.md](bias-checklist.md) を参照

### 1.3 公式ドキュメント・外部情報の収集

`WebSearch` で以下を検索:
- 関連する公式ドキュメント、API リファレンス
- 類似 Issue や Stack Overflow の質問
- RFC、仕様書（プロトコルやフォーマットの問題の場合）
- 関連するブログ記事やチュートリアル

`WebFetch` で特に重要なページの詳細内容を取得。

## 出力テンプレート

以下の形式で構造化サマリーを提示する:

```markdown

### 問題定義
- **Issue**: {タイトル} ({URL})
- **問題の本質**: {1文で表現}
- **分類**: [バグ | 機能追加 | パフォーマンス | リファクタリング | セキュリティ]
- **再現条件**: {わかっている場合}

- **問題箇所**: `{ファイルパス}:{行番号}` — `{関数/メソッド名}`
- **関連コード**:
  - `{ファイル1}` — {役割}
  - `{ファイル2}` — {役割}
- **依存関係**:
  ```
  [問題箇所] → [依存先1] → [依存先2]
                ↓
              [影響先1]
  ```

### テスト状況

- **既存テストカバレッジ**: {問題箇所に対するテストの有無と内容}

| {claim2} | structural | {score} | {反証結果} | {判定} |
| {claim3} | historical | {score} | {反証結果} | {判定} |

**仮説間の関係**:

```

## 確認ゲート

サマリーを提示した後:
1. 「この分析に認識違いや追加で調査すべき観点はありますか？」と確認
2. ユーザーの「OK」で Phase 2 へ進む
3. 追加調査の要望があれば対応してから再度確認
