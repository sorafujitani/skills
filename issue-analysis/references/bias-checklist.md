# バイアス・ハルシネーションチェックリスト

Phase 1 の出力を確定する前に、以下のセルフチェックを実行する。
1つでも「いいえ」がある場合、該当する防止メカニズムを再実行する。

## チェック項目

### 認知バイアス

| バイアス | セルフチェック質問 | 防止メカニズム |
|---------|------------------|--------------|
| 確証バイアス | この結論に反する証拠を積極的に探したか？ | 独立並列調査 + 強制反証 |
| アンカリング | 最初の仮説と最終結論が同じ場合、証拠スコアで最も支持されているか？ | N-of-1ルール + 証拠スコアリング |
| 可用性ヒューリスティック | 「よくある原因だから」ではなく「このコードの証拠が示すから」と言えるか？ | 証拠強度スコアリング |
| フレーミング効果 | 報告者が述べていない箇所も調査したか？ | structural-analysis視点 |

### ハルシネーション防止

| チェック項目 | セルフチェック質問 | 検証方法 |
|------------|------------------|---------|
| ソース実在確認 | 引用したファイルパス・関数名・行番号は実際に存在するか？ | `Read` / `Grep` で該当箇所を再確認 |
| コード内容の正確性 | 引用したコードの内容は実際のコードと一致するか？ | `Read` で該当行を読み直して照合 |
| API・仕様の正確性 | 言及したAPI仕様・ライブラリの挙動は正しいか？ | `WebSearch` で公式ドキュメントを確認 |
| バージョン整合性 | 参照したドキュメントのバージョンはプロジェクトの依存バージョンと一致するか？ | package.json / go.mod 等と照合 |
| URL実在確認 | 出力に含めるURLは実在するか？ | `WebFetch` でアクセス可能性を検証 |
| 推論と事実の区別 | 事実として述べた内容にコード証拠があるか？推論なら「推論」と明記したか？ | 各主張に証拠タグ（direct_code / inference）を付与 |

## バイアス発生パターンと対策

### 確証バイアス
- **発生**: 最初に見つけた原因を支持する情報ばかり集める
- **対策**: 各subagentが独立に仮説を生成（情報共有なし）→ 反証agentが各仮説を攻撃
- **再実行条件**: 反証調査を1つも実施していない場合

### アンカリングバイアス
- **発生**: 最初の仮説に引きずられて他の可能性を過小評価する
- **対策**: 仮説が1つしかない場合は追加調査を強制（N-of-1ルール）
- **再実行条件**: 仮説が1つのみで追加調査を実施していない場合

### 可用性ヒューリスティック
- **発生**: 思い出しやすい（よくある）原因を過大評価する
- **対策**: 証拠強度スコアリングで客観評価。「よくある」は加点対象にしない
- **再実行条件**: 最有力仮説の証拠が inference_only のみの場合

### フレーミング効果
- **発生**: issue報告者の表現に引きずられて問題の範囲を限定する
- **対策**: structural-analysis視点が報告者の記述に依存せず構造的弱点を探す
- **再実行条件**: 全仮説が報告者の記述範囲内に限定されている場合

### ハルシネーション（幻覚生成）
- **発生**: 存在しないファイル・関数・API仕様を事実として出力する
- **対策**:
  1. **ソースチェック**: 出力に含まれるすべてのファイルパス・関数名を `Read` / `Grep` で実在確認
  2. **Web裏取り**: API仕様やライブラリの挙動に言及する場合、`WebSearch` で公式ドキュメントを取得し照合
  3. **証拠タグ付与**: 各主張に `direct_code` / `test` / `git_history` / `doc` / `inference` のいずれかを明記。`inference` のみの主張は「推論」と明示
- **再実行条件**: ファイルパスや関数名を `Read` で確認していない場合
