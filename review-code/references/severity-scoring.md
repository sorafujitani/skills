# 重大度分類と評価基準

## 重大度定義

### 🚨 Critical（修正必須）
- 実行時エラー、データ不整合、セキュリティ脆弱性を引き起こす
- 本番環境で障害やデータ損失のリスクがある
- マージ前に必ず修正が必要

### ⚠️ Important（修正推奨）
- 将来的な問題やパフォーマンス低下につながる可能性
- コードの保守性・可読性を著しく損なう
- テスト不足により品質保証が不十分
- マージ前の修正を強く推奨

### 💡 Minor（提案）
- より良い実装方法やベストプラクティスの提案
- 命名やフォーマットの改善
- リファクタリングの機会
- マージには影響しない

## 証拠ベース評価

各指摘は以下の基準を満たすこと:

1. **具体的な場所**: `file:line` で特定
2. **コード証拠**: 問題のあるコードを引用
3. **実在確認**: 指摘箇所を `Read` で再確認済み
4. **理由説明**: なぜ問題なのかを技術的に説明
5. **改善案**: 具体的な修正方法を提示

## ハルシネーション防止

指摘を出力する前に必ず検証する:

| チェック | 方法 |
|---------|------|
| ファイル・行番号の実在 | `Read` で該当行を確認 |
| 引用コードの正確性 | `Read` の出力と照合 |
| API仕様の正確性 | `WebSearch` で公式ドキュメントを確認 |
| 推論と事実の区別 | 推論のみの指摘は「可能性がある」と明記 |

## 指摘の品質基準（obra/superpowers準拠）

### 良い指摘の条件
- 正確な重大度分類
- `file:line` 参照付き
- 理由の明確な説明
- 強みの承認も含む

### 避けるべき指摘
- 漠然とした評価（「もう少し良くできそう」）
- 過度な重大度付け（minor を critical にしない）
- 未確認のフィードバック（コードを読まずに推測で指摘）
- 曖昧な改善提案（「リファクタリングすべき」だけ）

## YAGNI 適用基準

レビューで「将来のために」と提案する場合:
- 現在のPRスコープで本当に必要か確認
- 使われていない機能の洗練は pushback する
- プロジェクトの実際の使用パターンに基づいて判断

## マージ判定

最終評価で以下を明示:
- **LGTM**: Critical/Important なし → マージ可
- **要修正**: Critical あり → 修正後に再レビュー
- **要検討**: Important のみ → 修正推奨だが判断はオーナーに委ねる
