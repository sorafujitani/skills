# 7次元設計評価ルーブリック

各次元 1-5点。合計 /35。

## スコア分類

| 合計 | 評価 | アクション |
|------|------|-----------|
| 28-35 | Excellent | そのまま実装可能 |
| 21-27 | Good | 軽微な改善で実装可能 |
| 14-20 | Acceptable | 弱い次元の対策を追加してから実装 |
| <14 | Redesign | アプローチの再検討が必要 |

## 次元定義

### D1: Testability

```
5: 全コンポーネントがDIで分離可能。pure function中心。境界が明確でseamが自然に存在
4: 主要コンポーネントのテスト容易。一部にハードコード依存あるが分離可能
3: テスト可能だがモック作成にコストがかかる。前処理が多い
2: 密結合箇所あり。テストにはリファクタが先に必要
1: グローバル状態依存、外部依存の分離困難、非決定的要素多数
```

**チェック項目**: DI可能性 / pure function比率 / テストseam / 外部依存の分離

### D2: Changeability

```
5: OCP準拠。拡張ポイントが明示的。変更が局所的に収まる
4: 主要な変更パターンに対して安定。一部shotgun surgery
3: 変更可能だが複数ファイル修正が必要
2: 変更が波及的。影響範囲の予測が困難
1: 密結合。変更のたびに回帰テスト全面必要
```

**チェック項目**: 変更局所性 / 拡張ポイント / 依存方向の安定性

### D3: Pattern Appropriateness

```
5: パターンが問題に自然適合。過剰設計なし。コードベースの既存パターンと整合
4: パターンは適切。若干のボイラープレート
3: パターンは機能するが一部over-engineering
2: パターンの選択に疑問あり。simpler alternativeが存在
1: パターンが問題にミスマッチ or ad-hoc実装
```

**チェック項目**: 既存パターンとの整合 / 複雑さの妥当性 / ボイラープレート量

### D4: Language/Runtime Fit

```
5: 言語イディオムを最大活用。エコシステムに最適化。型システム活用
4: 言語の主要機能を適切に使用
3: 動作するが言語固有の利点を活かしきれていない
2: 他言語パターンの移植で不自然な箇所あり
1: 言語の特性に反した設計。アンチパターン使用
```

**チェック項目**: イディオム適合 / 型安全性 / エコシステム活用 / ランタイム特性考慮

### D5: Dependency Management

```
5: 依存方向明確。循環なし。外部依存最小。安定依存の原則準拠
4: 概ね良好。依存グラフが理解可能
3: 一部の依存が不明瞭だが管理可能
2: 循環依存の兆候。外部依存が過多
1: 循環依存。バージョン競合リスク。不要な外部依存
```

**チェック項目**: 依存方向 / 循環有無 / 外部依存の妥当性 / 安定依存の原則

### D6: Error Handling Strategy

```
5: エラー型体系化。回復可能/不可能が区別。ユーザーへの適切なフィードバック
4: 主要パスのエラー処理完備。型安全
3: 基本的なエラー処理あり。一部catch-all
2: エラー処理が不完全。サイレント失敗のリスクあり
1: エラー処理が場当たり的。未処理の例外パスあり
```

**チェック項目**: エラー型設計 / 回復戦略 / ログ/通知 / ユーザーフィードバック

### D7: Performance Considerations

```
5: ホットパス最適化。計算量明示。ボトルネック予測可能。キャッシュ戦略明確
4: 主要なパフォーマンス考慮あり。明らかなボトルネックなし
3: 基本的な性能考慮あり。最適化の余地が明確
2: N+1やメモリリーク等の潜在リスクあり
1: 性能考慮なし。明らかなボトルネック存在
```

**チェック項目**: 計算量 / メモリ使用 / I/O最適化 / キャッシュ / 並行処理

## ウェイト調整（オプション）

ユーザーが特定の次元を重視する場合、確認ゲートで重みを調整可能:

```
default:  均等 (各1.0x)
api-heavy: D6=1.5x, D7=1.5x, D3=0.75x
library:   D2=1.5x, D4=1.5x, D1=1.25x
prototype: D3=0.5x, D7=0.5x, D1=1.5x
```
