# 前提知識の提供ガイドライン

## 核心原則

**概念説明だけのステップを作らない。print 指示の文脈として1-2文で添える。**

## やり方: print 指示に織り込む

### ❌ 概念説明が独立したステップになっている

```
Step 1: 「パーサーは文字列をトークンに分解し、ASTに変換します。」→ ok
Step 2: 「intrinsic は括弧の有無でノード種別が変わります。」→ ok
Step 3: 「では parse_type() に print を入れてみましょう」→ 実行
```

### ✅ 概念が print 指示の一部になっている

```
Step 1: 「パーサーが intrinsic をどう解釈するか確認しましょう。
         括弧があると別のノードになる可能性があります。
         parse_type() の先頭にトークン種別を出力する print を入れてください。」→ 実行
```

## 判断基準: この説明は print に必要か？

print 指示を出すとき、以下の場合だけ概念を添える:

1. **print の出力を解釈するのに必要** — 「Debug 出力に TSTypeReference と出たら括弧ありの intrinsic、TSIntrinsicKeyword と出たら括弧なしの intrinsic です」
2. **print の場所を理解するのに必要** — 「各ノードが gen() で自身を出力するので、TSIntersectionType::gen() に print を入れます」
3. **なぜそこに print を入れるのかの理由** — 「パーサーはトークン種別で分岐するので、分岐の手前で今のトークンを出力します」

以下は添えない:
- ユーザーが既に知っていること
- print の実行に関係ない背景知識
- 「あとで分かります」で後回しにできること

## プロジェクト固有の慣習

print を入れる場所に関係する場合だけ1文で伝える:

- 「このファイルは自動生成ですが、一時的な print は OK です」
- 「この enum は #[derive(Debug)] がついているので {:?} で出力できます」

## アンチパターン

### 講義モード
```
❌ 「まずパーサーの仕組みを説明します。パーサーは...（3パラグラフ）」
✅ 「parse_type() に print を入れて、パーサーが何を見ているか確認しましょう」
```

### 説明なしの print 指示
```
❌ 「statement.rs:136 に println! を入れてください」
✅ 「パーサーがトークン種別で分岐するので、statement.rs:136 の分岐前に
   今のトークン種別を出力してください」
```

両極端を避ける。print 指示 + 「なぜここか」の1文 が適切な粒度。
