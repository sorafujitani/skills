# 観察ポイントの設計原則

## 配置戦略

### 二分探索アプローチ

パイプラインの中間点から始め、問題のあるステージを絞り込む:

```
ステージ1 → ステージ2 → ステージ3 → ステージ4
                  ↑
            最初のprint（中間点）
```

- 中間点で正常 → 問題は後半のステージ
- 中間点で異常 → 問題は前半のステージ

### 境界観察

ステージの入口と出口の両方を観察する:

```
ステージN:
  [入口] → 入力データを出力
  ... 処理 ...
  [出口] → 出力データを出力
```

入口は正常だが出口で異常 → このステージ内に問題がある

### 分岐観察

条件分岐のどちら側に入ったかを観察する:

```
if condition {
    // [print A] ← ここを通った？
} else {
    // [print B] ← それともここ？
}
```

## 出力すべき情報の選び方

### 最小限から始める

1. **まず通過確認**: `[DEBUG] ここを通った` — コードパスの確認だけ
2. **次に値の確認**: `[DEBUG] x = {:?}` — 変数の中身
3. **最後に詳細**: `[DEBUG] full context = {:?}` — 周辺情報

### 言語別のprint方法

**Rust**:
```rust
println!("[DEBUG] 場所: 変数 = {:?}", variable);
// Debug trait未実装の場合:
println!("[DEBUG] 場所: 通過");
```

**TypeScript/JavaScript**:
```typescript
console.log("[DEBUG] 場所:", variable);
console.log("[DEBUG] 場所:", JSON.stringify(variable, null, 2));
```

**Python**:
```python
print(f"[DEBUG] 場所: {variable=}")
```

**Go**:
```go
fmt.Printf("[DEBUG] 場所: %+v\n", variable)
```

### ラベルの付け方

複数のprint文を区別するためにラベルを付ける:

- プレフィックスで分類: `[PARSE]`, `[FORMAT]`, `[EMIT]`
- 絵文字で視認性向上: `🟥`, `🟦`, `🟩`（ユーザーが使いたい場合）
- 番号で順序追跡: `[1]`, `[2]`, `[3]`

## 観察ポイント数の目安

- **最初のイテレーション**: 2-3箇所（多すぎると混乱する）
- **深掘りイテレーション**: 1-2箇所（焦点を絞る）
- **確認イテレーション**: 1箇所（仮説の最終検証）

## print文が通らなかった場合

「print文が表示されない」は重要な情報:

1. **コードパスが通っていない** → 想定と異なる分岐に入っている
2. **関数自体が呼ばれていない** → 呼び出し元を確認する必要がある
3. **出力がバッファリングされている** → flush の追加を検討

いずれも学習機会として活用する。
